# Alembic - Database Migrations

Alembic handles database schema changes via version-controlled migrations. No more `SQLAlchemy.metadata.create_all()`.

## Setup (already done)

- `alembic/` directory initialized
- `alembic.ini` configured with PostgreSQL connection
- `alembic/env.py` imports models for autogenerate
- Initial migration: `alembic/versions/001_initial_schema.py`

## Working with Migrations

```
# init alembic
source venv/bin/activate && python -m alembic init alembic
```

### 1. Create new migration after model changes

```bash
python -m alembic revision --autogenerate -m "Add column to tasks"
```

Requires database connection. Creates file in `alembic/versions/`.

**Note**: `--autogenerate` needs running DB. For CI/CD, migrations are created locally and committed.

### 2. Review migration file

Generated migrations in `alembic/versions/`:

```python
def upgrade() -> None:
    # Apply change
    op.add_column('tasks', sa.Column('priority', sa.Integer()))

def downgrade() -> None:
    # Rollback change
    op.drop_column('tasks', 'priority')
```

Always review autogenerated code — AI-generated migrations sometimes need tweaks.

### 3. Apply migrations to database

```bash
# Upgrade to latest
python -m alembic upgrade head

# Upgrade to specific revision
python -m alembic upgrade 002

# Show current revision
python -m alembic current

# Show migration history
python -m alembic history
```

### 4. Rollback migration

```bash
# Down one migration
python -m alembic downgrade -1

# Down to specific revision
python -m alembic downgrade 001
```

## Docker / Production

Migrations run automatically on container startup:

```bash
docker-compose up
# Container runs: alembic upgrade head → python -m app.main
```

See `docker-entrypoint.sh` — runs migrations before starting app.

## CI/CD Integration

In GitHub Actions or other CI:

```bash
export DATABASE_URL="postgresql://user:pass@host:5432/db"
python -m alembic upgrade head
pytest
```

Database must be ready (Postgres service running) before running migrations.

## Writing Migrations Manually

If autogenerate misses something or you want full control:

```bash
python -m alembic revision -m "Custom migration"
```

Creates empty template. Edit `upgrade()` and `downgrade()` using `op.*`:

```python
from alembic import op
import sqlalchemy as sa

def upgrade() -> None:
    op.create_table(
        'new_table',
        sa.Column('id', sa.Integer(), nullable=False),
        sa.Column('name', sa.String(), nullable=False),
        sa.PrimaryKeyConstraint('id')
    )

def downgrade() -> None:
    op.drop_table('new_table')
```

See [Alembic Operations](https://alembic.sqlalchemy.org/en/latest/ops.html) for all operations.

## Best Practices

1. **Commit migrations to git** — they're part of your codebase
2. **One migration per logical change** — easier to understand and rollback
3. **Test migrations locally** before pushing:
   ```bash
   python -m alembic upgrade head
   # Test app
   python -m alembic downgrade -1
   python -m alembic upgrade head  # Verify it works both ways
   ```
4. **Never edit committed migrations** — create new ones to fix issues
5. **Descriptive messages** — "Add user authentication" not "Fix schema"

## Workflow

### Local development

```bash
# Make model changes in app/models.py
# Generate migration
python -m alembic revision --autogenerate -m "Add email to users"
# Review alembic/versions/002_*.py
# Test migration
python -m alembic upgrade head
# Run tests
pytest
# Commit
git add alembic/versions/002_*.py
git commit -m "feat(db): Add email to users"
```

### Before merging to main

- All migrations committed
- Tested locally (upgrade + downgrade)
- No uncommitted schema changes in models

### On deployment

- Docker runs migrations automatically
- CI verifies migrations apply without error
- Rollback: `alembic downgrade -1` (keep this command documented)

## Troubleshooting

### "No such revision"

Migration file missing or not committed:

```bash
ls alembic/versions/
git status alembic/
```

### "Migration conflicts"

Two migrations created for same revision number. Rename one:

```bash
mv alembic/versions/002_*.py alembic/versions/003_*.py
# Edit file: change revision = "003"
```

### "Database locked" (SQLite only)

Not relevant for PostgreSQL. This happens on SQLite when migrations run concurrently.

### Manual fix needed

If migration partially applied:

```bash
# Mark migration as applied without running
python -m alembic stamp 002

# Or mark as unapplied
python -m alembic downgrade -1
```

## Environment Variables

Alembic reads `DATABASE_URL` from environment:

```bash
# Local (config.env loaded by app)
python -m alembic upgrade head

# Docker
export DATABASE_URL="postgresql://taskuser:taskpass@db:5432/taskdb"
python -m alembic upgrade head
```
